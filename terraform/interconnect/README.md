# AWS ↔ Azure Site-to-Site VPN

BGP-routed IPsec VPN between the default VPC in an AWS region and an
existing Azure VNet. No native peering exists between AWS and Azure, so
this uses each cloud's managed VPN gateway with dynamic routing — routes
propagate automatically in both directions, no static route maintenance.

```
 AWS default VPC                                Azure VNet
 ┌─────────────────┐                        ┌──────────────────────┐
 │ Virtual Private │  tunnel 1 (IPsec/BGP)  │  VPN Gateway         │
 │ Gateway         │========================│  (VpnGw1AZ,          │
 │ (ASN 64512)     │  tunnel 2 (IPsec/BGP)  │   ASN 65515)         │
 └─────────────────┘========================└──────────────────────┘
        │ route propagation                        │ automatic
        ▼                                          ▼
   VPC route tables                          VNet routes
```

One AWS VPN connection provides two tunnels (distinct AWS endpoints) for
redundancy; both terminate on the Azure gateway. BGP sessions run over
APIPA /30s inside 169.254.21.0/24 and 169.254.22.0/24 — the only ranges
Azure accepts for custom BGP peering addresses, and both outside AWS's
reserved inside-CIDR list.

## Prerequisites

- AWS credentials (`AWS_PROFILE` or SSO) with VPN/VPC permissions.
- Azure credentials (`az login` or `ARM_*` env vars) with network
  contributor on the target resource group.
- The VPC CIDR and VNet address space must not overlap.
- A free /27 in the VNet for the `GatewaySubnet`.

## Usage

This directory is a module; provider configuration lives in the repo
root `main.tf`, which is the plan/apply entry point. Defaults target
the ybor-playground environment (AWS profile
`ybor-playground-dev@p6m.dev`, Azure subscription `ybor-playground`,
VNet `ybor-playground-dev-westus2`).

```sh
# from the repo root
aws sso login --profile ybor-playground-dev@p6m.dev
az login
terraform init
terraform plan
terraform apply   # Azure VPN gateway takes ~30-45 minutes to create
```

Tunnel pre-shared keys are generated by AWS and wired into the Azure
connections automatically; they live in Terraform state, so use an
encrypted remote backend before running this anywhere shared.

## Verify

```sh
# Both tunnels should reach UP within a few minutes of apply
aws ec2 describe-vpn-connections --vpn-connection-ids $(terraform output -raw aws_vpn_connection_id) \
  --query 'VpnConnections[0].VgwTelemetry[].Status'

# BGP peers should show Connected with routes learned
az network vnet-gateway list-bgp-peer-status \
  --resource-group <rg> --name <prefix>-vpngw -o table
```

## Costs (approximate, monthly)

| Item | Cost |
|---|---|
| Azure VPN Gateway (VpnGw1AZ) | ~$260 |
| AWS Site-to-Site VPN (2 tunnels, $0.05/hr) | ~$36 |
| Data transfer | egress rates both sides |

Throughput ceiling: ~650 Mbps aggregate (VpnGw1AZ), ~1.25 Gbps per tunnel
on the AWS side. Scale the Azure SKU via `azure_vpn_gateway_sku`.

## Limitations

- Azure gateway is active-standby with a single public IP. Full
  active-active HA needs two public IPs and a second AWS VPN connection.
- For >1 Gbps sustained or private-path requirements, move to
  Direct Connect + ExpressRoute via a NaaS provider (Megaport, Equinix
  Fabric) instead.
